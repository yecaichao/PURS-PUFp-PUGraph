# Paper Reproduction Matrix

## Scope

This note maps the current unified repository to the two paper workflows that matter most for the
project:

- `PUFp` paper:
  - `54. PUFp_acsami.3c03298.pdf`
- `PUGraph` paper:
  - `67. JCTC-zhang-et-al-2024-polymer-unit-graph-advancing-interpretability-in-graph-neural-network-machine-learning-for-organic.pdf`

The goal is not only to summarize the papers, but to define which repository modules, datasets, and
tests are needed to reproduce their core claims.

## Paper A: PUFp (ACS AMI 2023)

### Main workflow in the paper

The paper describes a six-step workflow:

1. collect an OSC polymer dataset
2. identify polymer-units using `PURS`
3. build a dynamic polymer-unit library
4. convert each polymer into a one-dimensional `PUFp`
5. train traditional ML models on mobility-related tasks
6. analyze important polymer-units and polymer-unit combinations

### Dataset and targets

Based on the main text extracted from the PDF:

- the dataset contains `678` experimentally reported polymer OSC materials
- each sample contains at least:
  - polymer `SMILES`
  - electron mobility
  - hole mobility
- the paper defines:
  - `n-type` if electron mobility > hole mobility
  - `p-type` otherwise
- mobility is further grouped into:
  - `high`: `> 4 cm2 V-1 s-1`
  - `medium`: `1-4 cm2 V-1 s-1`
  - `low`: `0-1 cm2 V-1 s-1`

### PUFp construction details

The paper states:

- polymer-units are identified by `PURS`
- a dynamic polymer-unit library is built from the dataset itself
- the final fingerprint is generated by indexing each polymer against the polymer-unit library
- a node matrix is first constructed, then summed into a one-dimensional vector
- the resulting `PUFp` length in the paper is `445 bits`

### Polymer-unit recognition rules explicitly stated in the paper

The main text lists these operational rules:

1. divide molecules at breakpoints
2. breakpoints must be single bonds connecting independent units
3. acyclic segments are classified as chains; chain fragments shorter than five atoms should merge into neighboring ring units
4. all polymer-unit SMILES should be canonicalized with `RDKit`

### ML models used in the paper

The main text explicitly names:

- `Random Forest`
- `KNN`
- `Gaussian Process`
- `SVM`
- `MLP`

### Reported outcomes to target in reproduction

The main text reports:

- overall classification accuracy reaches `85.2%`
- the polymer-unit library contains `445 units`

### What the repository should reproduce for this paper

Minimum reproducible targets:

1. `PURS -> polymer-unit library -> PUFp`
2. `n/p` classification from electron vs hole mobility
3. `high/medium/low` mobility classification
4. comparison against common fingerprints such as `MACCS` and `Morgan`
5. importance analysis at the polymer-unit level

### Current repository alignment

Already aligned:

- `purs recognize`
- `purs classify`
- `purs fingerprint`
- `purs ml rf|krr|svm`

Still missing for closer paper reproduction:

- explicit `n/p` classification target builder
- explicit `high/medium/low` classification target builder
- paper-style benchmark script comparing `PUFp` vs `MACCS` vs `Morgan`
- importance-analysis output in a paper-facing format
- exact `KNN`, `Gaussian Process`, and `MLP` baselines

## Paper B: Polymer-Unit Graph (JCTC 2024)

### Main workflow in the paper

The paper defines Polymer-Unit Graph generation in three steps:

1. identify polymer-units and their connectivity using `PURS`
2. create a polymer-unit database and assign graph-embedding node features
3. build Polymer-Unit Graphs with polymer-units as nodes and inter-unit bonds as edges

### Dataset and targets

From the main text:

- the dataset contains `697` OSC data points
- each sample includes:
  - polymer `SMILES`
  - electron mobility
  - hole mobility
  - `HOMO`
  - `LUMO`

### Node and edge features

The main text explicitly states:

- node embeddings tested:
  - `MACCS` code: `166 bits`
  - `GenWL` graph embedding: `280 bits`
- edge features are `8 x 1` vectors with three parts:
  - indices of connected polymer-units in the library
  - atomic numbers of connected atoms plus bond length
  - stereochemical one-hot information

### Models and analysis

Two graph branches are emphasized:

1. `PU-gn-exp`
   - built by replacing the input part of the baseline `gn-exp`
   - uses `166-bit MACCS` as node features
   - trained as a classification model
   - analyzed with `LRP`

2. `PU-MPNN`
   - built by combining Polymer-Unit Graph with a baseline `mol-MPNN`
   - focuses on improving efficiency while preserving prediction quality

### Reported outcomes to target in reproduction

The main text reports:

- `81.96%` classification accuracy for hole mobility visual-analysis model
- `88.20%` classification accuracy for electron mobility visual-analysis model
- `PU-MPNN` reduces training time by `98%`
- equivalent runtime is about `0.02` of the baseline `mol-MPNN`
- prediction quality is compared using `MSE`

### Important note on labels

The paper states that `PU-gn-exp` was transformed into a classification model and that carrier
mobility labels were divided into `four groups`, but the exact grouping rule is referenced via the
supporting information and `Table 1` rather than fully spelled out in the main text.

This means a faithful paper reproduction should capture the exact label transform from the
supporting information before claiming final reproduction.

### What the repository should reproduce for this paper

Minimum reproducible targets:

1. `PURS -> Polymer-Unit Graph`
2. node embeddings with at least:
   - `MACCS166`
   - optional `GenWL280`
3. `PU-gn-exp` style classification on:
   - hole mobility
   - electron mobility
4. `PU-MPNN` vs baseline `mol-MPNN` comparison on:
   - `MSE`
   - runtime
5. visualization and relevance attribution at the polymer-unit level

### Current repository alignment

Already aligned:

- `purs graph build`
- wrapped `PU-gn-exp`
- wrapped `PU-MPNN`
- legacy graph adapter already uses `MACCS` internally

Still missing for closer paper reproduction:

- explicit paper-style classification label transform for `PU-gn-exp`
- paper-style hole/electron mobility tasks as first-class configs
- `GenWL280` node-feature pipeline
- baseline `mol-MPNN` comparison harness
- runtime benchmark script
- paper-style relevance visualization export

## Unified repository design implied by both papers

The two papers actually fit one clean architecture:

1. shared data layer
2. shared `PURS` recognition layer
3. two feature branches
4. two model branches
5. one paper-facing evaluation layer

### Shared data layer

For faithful reproduction, the repository should maintain a paper-style OSC parent table with:

- `sample_id`
- `smiles`
- `ue`
- `uh`
- optional `homo`
- optional `lumo`
- optional `source_paper`
- optional `device_type`

### Shared PURS layer

This part should remain the common preprocessing trunk:

- polymer-unit identification
- polymer-unit connectivity
- polymer-unit library generation

### Feature branch A: PUFp-style

- `PUFp`
- optional `MACCS`
- optional `Morgan`
- optional fused descriptors such as `HOMO/LUMO/Eg`

### Feature branch B: PUGraph-style

- Polymer-Unit Graph
- node features:
  - `MACCS166`
  - optional `GenWL280`
- edge features:
  - current wrapped graph schema
  - later aligned more explicitly with the paper

### Model branch A: traditional ML

- `RF`
- `KNN`
- `Gaussian Process`
- `SVM`
- `MLP`

### Model branch B: graph ML

- `PU-gn-exp`
- `PU-MPNN`
- optional paper baseline wrappers:
  - `gn-exp`
  - `mol-MPNN`

## Recommended repository test hierarchy

### Level 1: smoke tests

Goal:

- verify every command path still works

Examples:

- tiny fixtures
- tiny `ue` subset
- tiny `PCE` subset

### Level 2: paper-style regression tests

Goal:

- verify paper-specific target transforms and feature generation

Examples:

- `n/p` label builder
- `high/medium/low` mobility bin builder
- Polymer-Unit Graph node-feature dimensions
- `MACCS166` and `GenWL280` integrity checks

### Level 3: paper reproduction runs

Goal:

- generate paper-facing outputs on the near-full datasets

Examples:

- PUFp benchmark suite
- PU-gn-exp paper task suite
- PU-MPNN runtime/MSE benchmark suite

## Recommended implementation order

1. Create a canonical paper-style OSC parent dataset schema.
2. Add a `repro/pufp_acsami_2023/` pipeline:
   - target builders
   - fingerprint baselines
   - ML comparison script
3. Add a `repro/jctc_2024_pugraph/` pipeline:
   - graph node-feature builders
   - label transform builder
   - `PU-gn-exp` and `PU-MPNN` scripts
4. Add a shared evaluation summary format so both paper reproductions emit comparable JSON and CSV outputs.
5. Pull exact hyperparameters and class-bin definitions from the supporting information before finalizing any claim of full reproduction.

## Practical conclusion

For the unified program, the right goal is not to choose between the two papers.

Instead, the repository should treat them as two complementary heads on top of the same shared
`PURS` preprocessing trunk:

- `Paper A`: `PURS -> PUFp -> traditional ML`
- `Paper B`: `PURS -> Polymer-Unit Graph -> GNN`

That shared design is exactly what makes these papers suitable as repository-scale regression and
reproduction tests.
